## 日志规范

编写代码时，你必须严格遵循以下日志打印原则与格式规范，确保日志兼顾人类可读与 AI 可解析。

### 打印原则

1. **外部调用必打**：调用外部模块或服务前打印请求参数，调用后打印返回结果与耗时
2. **关键流程节点与状态变更必打**：业务流程的起止、状态机转换、任务完成等关键节点必须记录
3. **异常错误必打**：必须包含错误码、错误描述、根因线索
4. **防止日志暴打**：禁止在高频循环中逐条打印日志，禁止打印大段二进制数据。高频调用场景（如缓存读取）正常成功时不打印日志，仅在异常时记录
5. **防止敏感信息泄露**：禁止在日志中输出密码、认证凭据、加解密密钥、Token 等敏感信息

### 日志等级标准

严格按以下定义使用日志等级，禁止混用：

**ERROR** — 影响业务功能正常运行的错误，需要人工介入：
- 外部调用失败且无法自动恢复（重试耗尽、连接彻底断开）
- 数据一致性问题（写入失败、校验不通过）
- 系统级异常（资源耗尽）
- 必须包含：错误码、错误描述、根因线索、关联参数

**WARN** — 异常但可自动恢复，不影响主流程：
- 外部调用失败但会自动重试
- 降级处理生效
- 配置缺失使用了默认值
- 接近阈值警告
- 必须包含：异常描述、当前状态、恢复策略

**INFO** — 关键业务流程节点与状态变更，用于链路追踪：
- 服务启动/停止
- 外部调用成功（含结果摘要与耗时）
- 关键业务状态变更（任务开始/完成、状态机转换）
- 配置加载完成
- 必须包含：事件描述、关键参数

**DEBUG** — 开发调试阶段的详细信息，生产环境默认关闭：
- 函数入参/出参
- 中间计算过程
- 协议/编解码详细内容
- 必须包含：详细的上下文变量值

**等级使用规则**：
- 重试场景：中间每次失败打 WARN，最终成功打 INFO 并附带重试次数；最终失败打 ERROR
- 高频调用场景：正常成功不打日志，仅在异常时打 WARN/ERROR

### 结构化日志格式

每条日志必须由以下三部分构成，保持统一的文本格式：

1. **上下文 TAG**：根据实际项目与业务情况选取必要的 TAG 字段，不强制要求所有字段。常见字段包括但不限于：时间戳、日志等级、模块名、线程/协程ID、链路标识（traceID/sessionID/taskID）、类名:函数名
2. **文本描述**：用简洁英文描述事件或错误，如 `Connect to server failed`、`Task completed successfully`
3. **KV 参数**：以 `key=value` 形式附加详细参数，逗号分隔

**格式模板**：
```
[TAG1] [TAG2] ... [类名:函数名] 文本描述, key1=value1, key2=value2
```

**示例**：
```
[2026-04-07 10:23:45.678] [ERROR] [NetworkModule] [tid:1234] [traceID:abc-001] [ConnectionManager:connect] Connect to server failed, host=192.168.1.100, port=8080, errorCode=ETIMEDOUT, retryCount=3
[2026-04-07 10:23:46.100] [INFO] [NetworkModule] [tid:1234] [traceID:abc-001] [ConnectionManager:connect] Connect to server succeeded after retry, host=192.168.1.100, port=8080, retryCount=3, latency=422ms
```
